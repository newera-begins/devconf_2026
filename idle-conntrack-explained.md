# Idle Connection Conntrack Timeout — Deep Concept Explanation

## What This Document Covers

This explains Scenario 3 (The Expired Connection) — how conntrack timeout works,
why idle database connections break silently, why every tool blames the database
when the real culprit is the kernel, and how eBPF is the only way to identify it.

---

## The Problem: Connection Pools and Conntrack Timeouts

### How Connection Pools Work

Most applications use **connection pools** to databases. Instead of opening a new
TCP connection for every query, the app maintains a pool of pre-established
connections and reuses them:

```
Backend-API pod:
  ┌─────────────────────────────────────┐
  │  Connection Pool (e.g., psycopg2)   │
  │                                     │
  │  conn_1: ESTABLISHED → idle 2s     │ ← recently used
  │  conn_2: ESTABLISHED → idle 45s    │ ← idle for 45 seconds
  │  conn_3: ESTABLISHED → idle 120s   │ ← idle for 2 minutes
  │  conn_4: ESTABLISHED → idle 5s     │ ← recently used
  └─────────────────────────────────────┘
```

When a new request arrives, the pool hands out an idle connection. The problem
is: **the application thinks the connection is still open, but the kernel's
conntrack entry may have already expired.**

### The Conntrack Timeout

Every conntrack entry has a timeout. For ESTABLISHED TCP connections, the default
is `net.netfilter.nf_conntrack_tcp_timeout_established = 432000` (5 days).

But in some environments this gets lowered:
- Cloud provider load balancers often set idle timeouts to 60-350 seconds
- Security teams lower it to reduce conntrack table memory usage
- Some OCP tuning guides recommend lower values for high-connection workloads
- In our demo, we set it to 30 seconds to show the effect quickly

### What Happens When the Timeout Fires

```
Timeline:

  T=0s:    App sends SQL query via conn_2
           Kernel conntrack: ESTABLISHED, timeout resets to 30s
           Packet forwarded to database ✓

  T=1s:    Database responds with results
           Conntrack: still ESTABLISHED, timeout resets to 30s
           Response delivered to app ✓

  T=2-30s: Connection idle. No packets.
           Conntrack timer counting down: 28...20...10...5...

  T=30s:   ★ CONNTRACK ENTRY EXPIRES ★
           Kernel removes the entry from the conntrack table.
           The TCP connection is still "open" at the application level
           (both sides have TCP state ESTABLISHED in their socket).
           But the kernel NO LONGER tracks this connection.

  T=35s:   App sends another SQL query via conn_2
           Packet: src=backend-api:45678, dst=database:5432, flags=PSH,ACK

           Kernel conntrack evaluates:
             "I see a TCP packet with ACK flag and data."
             "Let me look up the connection in my table..."
             "No matching entry found."
             "A packet with ACK but no conntrack entry = INVALID"
             → Mark as CT_INVALID
             → DROP the packet

           App never gets a response.
           App's TCP stack retransmits → same thing, dropped again.
           Eventually: "connection reset by peer" or "broken pipe"
```

### Why New Connections Still Work

New connections start with a SYN packet. When conntrack sees a SYN with no
existing entry, it creates a NEW entry with state NEW → SYN_SENT. This is normal.
The issue only affects packets that reference a connection that conntrack has
FORGOTTEN.

```
New connection (T=36s):
  SYN from backend-api:55000 → database:5432
  → Conntrack: "New SYN, no existing entry → create NEW entry"
  → Entry created, state: NEW
  → Packet forwarded to database ✓
  → SYN-ACK comes back → state: ESTABLISHED ✓
  → Works fine until this entry also expires after 30s of idle
```

## Why Traditional Tools Blame the Database

### tcpdump

```
$ tcpdump -i any -n port 5432
14:50:35 IP 10.128.2.15.45678 > 10.128.3.22.5432: Flags [P.], seq 1:48, ack 1
    ↑ Data packet sent (SQL query on reused connection)

    ... nothing ... (packet was dropped by conntrack, never reached database)

14:50:36 IP 10.128.2.15.45678 > 10.128.3.22.5432: Flags [P.], seq 1:48, ack 1
    ↑ TCP retransmit (same packet, still dropped)

14:50:38 IP 10.128.3.22.5432 > 10.128.2.15.45678: Flags [R.]
    ↑ RST from the database? NO — this RST is generated by the kernel
      because the packet is CT_INVALID
```

**What you think:** "The database sent a RST. Something is wrong with PostgreSQL."
**What actually happened:** The kernel dropped the data packet (CT_INVALID) and
sent a RST because there's no valid conntrack entry. The database never even
SAW the data packet.

### Application Logs

```
psycopg2.OperationalError: server closed the connection unexpectedly
    This probably means the server terminated abnormally
    before or while processing the request.
```

**What you think:** "PostgreSQL crashed or terminated the connection."
**What actually happened:** The TCP connection was fine at the application level.
The kernel's conntrack entry expired, and the next packet was dropped. The app
interprets the resulting timeout/RST as "server closed the connection."

**Teams typically spend 1-3 days debugging PostgreSQL:**
- Check PostgreSQL logs (nothing unusual)
- Check max_connections (not exceeded)
- Check idle_in_transaction_session_timeout (not set)
- Check tcp_keepalives_idle (set to 0 = disabled)
- Increase various PostgreSQL timeouts
- **None of this helps because the problem is in the kernel, not PostgreSQL**

### conntrack -L

```
$ conntrack -L -s 10.128.2.15 -d 10.128.3.22
    (no entries)
```

**The entry expired. It's gone. There's nothing to find.**

You're looking for evidence that has been destroyed. The conntrack table only
shows CURRENT entries. Historical entries that timed out are gone forever —
unless something recorded the drop when it happened. Only eBPF does that.

## What NOO Shows

```
┌───────────────┬────────────────┬──────────┬────────┬────────────┬──────────────┐
│ Source         │ Destination    │ Node     │ Proto  │ Drop Count │ Drop Reason  │
├───────────────┼────────────────┼──────────┼────────┼────────────┼──────────────┤
│ backend-api   │ database       │ worker-1 │ TCP    │ 8          │ CT_INVALID   │
│ backend-api   │ database       │ worker-1 │ TCP    │ 5          │ CT_INVALID   │
└───────────────┴────────────────┴──────────┴────────┴────────────┴──────────────┘

Pattern:
  ★ Drops ONLY on backend-api → database flows
  ★ Drops happen ~30-35 seconds after the last successful packet
  ★ New connections (from frontend → backend-api) have ZERO drops
  ★ Drop reason: CT_INVALID (no matching conntrack entry)

Root Cause: conntrack ESTABLISHED timeout is too low (30s).
Fix: Increase nf_conntrack_tcp_timeout_established or configure
     application-level TCP keepalives to prevent idle connections.
```

**NOO gives you:**
1. WHICH flow is affected (backend-api → database, not frontend → backend)
2. The exact drop reason (CT_INVALID, not NetworkPolicy or routing)
3. The timing pattern (drops correlate with idle periods)
4. The node where drops happen

**This is impossible to get from tcpdump + conntrack -L** because:
- tcpdump shows the RST but doesn't know it's from CT_INVALID
- conntrack -L can't show entries that already expired
- The timing correlation (idle > 30s → drop) requires per-flow timestamped drop data
  that only eBPF provides

## Summary

| Aspect | Traditional Tools | NOO + eBPF |
|--------|------------------|------------|
| Symptom seen | "connection reset by peer" | CT_INVALID drops on specific flow |
| Who gets blamed | Database (PostgreSQL) | Kernel conntrack (timeout too low) |
| Debug time | 1-3 days (wrong direction) | 10 minutes (correct direction) |
| Evidence available | RST in tcpdump, empty conntrack -L | Per-flow drops with reason + timestamp |
| Root cause visible | No (entry expired, evidence gone) | Yes (kfree_skb captured the drop at the moment it happened) |
